# Relay Client Backend

Go API service for Relay's customer-facing account, organization, project, billing, and audit surfaces.

## Stack

- Gin for HTTP APIs
- Postgres for persistence
- Goose for migrations
- sqlc query definitions for typed database access
- Buf-generated gRPC contracts for internal service boundaries

## Local setup

```sh
cp .env.example .env
export DATABASE_URL="postgres://postgres:postgres@localhost:5432/relay?sslmode=disable"
goose -dir db/migrations postgres "$DATABASE_URL" up
buf generate
sqlc generate
go run ./cmd/api
```

For local development auth, send:

```http
X-Auth0-Sub: auth0|local-user
X-User-Email: you@example.com
X-User-Name: Your Name
```

JWT signature validation against Auth0 JWKS should replace the dev header path before production.

## Hypervisor management transport

Exos hypervisors are managed over two layers:

- WireGuard provides the private management network between the control plane and hypervisor hosts.
- The hypervisor agent opens a persistent outbound gRPC stream to the control plane. The control plane sends VM commands over that stream; the agent reports snapshots and command results back on the same connection.

On first boot, a hypervisor does not need pre-existing access to the private management network. It dials the public WireGuard gateway endpoint, such as `vpn-1.exos.tech:51820`, using the key material generated by the control plane and written by Tinkerbell or the manual installer. Once WireGuard is up, the agent connects to the private control-plane gRPC address, such as `172.200.0.1:9090`.

The database is the source of truth for WireGuard peer desired state:

- `wireguard_gateways` describes each VPN gateway interface, including endpoint, public key, and tunnel address. `vpn-1` should use `172.200.0.1/32`.
- `hypervisor_wireguard_peers` describes the desired peer public key, hypervisor tunnel IP, allowed IPs, and present/absent state for each hypervisor. Hypervisors should use `172.200.1.0/24`.
- `hypervisor_agent_credentials` tracks issued/revoked agent client certificates.
- `hypervisor_conversions` tracks the lifecycle of converting a physical server into a managed hypervisor host.

Run the gateway reconciler with:

```sh
WG_GATEWAY_ID=wg-control-plane-1 \
DATABASE_URL="postgres://postgres:postgres@localhost:5432/relay?sslmode=disable" \
go run ./cmd/wireguard-controller
```

In Kubernetes, use `deploy/wireguard-controller.yaml` as the starting manifest. It should run only on nodes that own the WireGuard gateway interface and needs `NET_ADMIN` plus host access to `/dev/net/tun`.

For production hypervisor gRPC, configure mTLS on the API:

```sh
HYPERVISOR_GRPC_TLS_CERT_FILE=/etc/exos/grpc/tls.crt
HYPERVISOR_GRPC_TLS_KEY_FILE=/etc/exos/grpc/tls.key
HYPERVISOR_GRPC_CLIENT_CA_FILE=/etc/exos/grpc/client-ca.crt
```

When client CA is configured, each hypervisor agent must present a client certificate whose subject common name or URI SAN `exos-hypervisor://<hypervisor_id>` matches the `hypervisor_id` it reports. The older `HYPERVISOR_AGENT_TOKEN` can still be used as an additional shared bootstrap gate, but certificate identity should be the durable production control.

The hypervisor image should set:

```sh
CONTROL_PLANE_GRPC_ENDPOINT=172.200.0.1:9090
CONTROL_PLANE_GRPC_TLS=true
CONTROL_PLANE_GRPC_AUTHORITY=bmc.exos.tech
CONTROL_PLANE_GRPC_CA_FILE=/etc/exos/pki/control-plane-ca.crt
HYPERVISOR_CLIENT_TLS_CERT_FILE=/etc/exos/pki/hypervisor.crt
HYPERVISOR_CLIENT_TLS_KEY_FILE=/etc/exos/pki/hypervisor.key
WG_PEER_ENDPOINT=vpn-1.exos.tech:51820
WG_PEER_ALLOWED_IPS=172.200.0.1/32
```

Provisioning still uses the rack-agent/Tinkerbell path. Hypervisor VM lifecycle commands use the hypervisor gRPC stream, and command state is persisted in `hypervisor_commands` so retries and command results are auditable.

### Install the hypervisor agent on an existing Ubuntu server

Use this path when a server is already running Ubuntu and does not need to be provisioned through Tinkerbell.

Build the hypervisor agent binary:

```sh
cd /path/to/relay/client-backend
CGO_ENABLED=0 GOOS=linux go build -o ../../scripts/exos-hypervisor-agent ./cmd/hypervisor-agent
```

Create an environment file from the example:

```sh
cd /path/to/relay/scripts
cp ../client-backend/deploy/hypervisor-agent.env.example ./hypervisor-agent.env
```

Fill in the host identity, control-plane, WireGuard, and SSH access values:

```sh
HYPERVISOR_ID=hv-ny-01
HYPERVISOR_HOSTNAME=hv-ny-01

CONTROL_PLANE_GRPC_ENDPOINT=172.200.0.1:9090
CONTROL_PLANE_GRPC_TLS=true
CONTROL_PLANE_GRPC_AUTHORITY=bmc.exos.tech
CONTROL_PLANE_GRPC_CA_FILE=/etc/exos/pki/control-plane-ca.crt
HYPERVISOR_CLIENT_TLS_CERT_FILE=/etc/exos/pki/hypervisor.crt
HYPERVISOR_CLIENT_TLS_KEY_FILE=/etc/exos/pki/hypervisor.key

WG_INTERFACE=wg0
WG_ADDRESS=172.200.1.1/32
WG_PRIVATE_KEY=<hypervisor-private-key>
WG_PEER_PUBLIC_KEY=<control-plane-public-key>
WG_PEER_ENDPOINT=vpn-1.exos.tech:51820
WG_PEER_ALLOWED_IPS=172.200.0.1/32

SSH_ALLOWED_CIDR=34.48.27.200/32
APPLY_HOST_NETWORKING=true
```

If mTLS is enabled, copy the control-plane CA and hypervisor client certificate/key onto the host:

```sh
sudo mkdir -p /etc/exos/pki
sudo cp control-plane-ca.crt /etc/exos/pki/
sudo cp hypervisor.crt /etc/exos/pki/
sudo cp hypervisor.key /etc/exos/pki/
sudo chmod 600 /etc/exos/pki/hypervisor.key
```

Run the installer on the Ubuntu server:

```sh
cd /path/to/relay/scripts
sudo AGENT_BINARY=./exos-hypervisor-agent \
  ENV_FILE=./hypervisor-agent.env \
  ./install-hypervisor-agent.sh
```

Restart and inspect the service:

```sh
sudo systemctl restart exos-hypervisor-agent
sudo systemctl status exos-hypervisor-agent
sudo journalctl -u exos-hypervisor-agent -f
```

The installer installs libvirt, QEMU, WireGuard, firewall dependencies, places the agent at `/usr/local/bin/exos-hypervisor-agent`, writes `/etc/exos/hypervisor-agent.env`, enables libvirt, and registers the `exos-hypervisor-agent` systemd service.

Before the agent can connect successfully, the control-plane database must contain a `wireguard_gateways` row for the selected VPN gateway, such as `vpn-1` at `172.200.0.1/32`, and matching WireGuard desired state for the hypervisor in `hypervisor_wireguard_peers`. The control-plane WireGuard controller will then allow the hypervisor tunnel IP, such as `172.200.1.1/32`, for that hypervisor public key.
